# Invariants — REPLACE_ME_COMPANY_NAME

Things that should be true across every decision, design choice, and
piece of work. When an invariant is broken, that's news worth surfacing.

This file is short on purpose. Invariants are durable; if you have a
hundred of them, they're not invariants.

## Example invariants

(These are starting examples. Edit, replace, or remove.)

### Be straight

Never sugarcoat status, never hide problems, never agree just to be
agreeable. If something is going wrong, say so immediately.

### YAGNI

Don't build what we don't need now. The best feature is the one that
gets used.

### Distribution beats features (once the product works)

Zero users means nothing else matters. Once a thing works, every hour
on more engineering instead of getting it in front of people is wasted.

### The team owns the outcome

Roles divide ownership so we can move without coordination overhead.
They don't divide responsibility for whether the company moves forward.

### Two voices on substantial work

A builder and a reviewer. The voices don't have to be different
agents, but they have to be different perspectives.

### Verify, don't speculate

When a question has an empirical answer (a test, a query, a smoke
check, asking the right person), get the empirical answer. Don't
default to assertion when verification is available.

## What's NOT in here

Tactical patterns (how to write a commit message; what tools to use)
belong in operating documents, not invariants. Invariants are the
above-the-line principles; the below-the-line implementations evolve.
